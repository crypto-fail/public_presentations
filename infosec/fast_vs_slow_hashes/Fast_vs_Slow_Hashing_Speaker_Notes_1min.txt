FAST VS. SLOW HASHING — SPEAKER NOTES (1 MINUTE PER SLIDE)
============================================================
Target: ~60 seconds per slide | ~10 minutes total


SLIDE 1 — Fast vs. Slow Hashing
--------------------------------
Welcome. Today we're covering a topic that sounds simple but trips up experienced engineers — when to use fast hashing algorithms versus intentionally slow ones. The subtitle says it all: when speed helps and when it hurts. Most developers know SHA-256 and bcrypt exist, but fewer understand why using the wrong one in the wrong context either cripples your application's performance or hands attackers your users' passwords. By the end of this deck, you'll have a clear framework for making that decision — which algorithm, which parameters, and which programming languages give you the best tools to handle sensitive data responsibly.


SLIDE 2 — Why Choosing the Right Hash Matters
----------------------------------------------
These three cards frame the entire presentation. On a single RTX 4090 GPU — a consumer card you can buy for $1,600 — an attacker can try 7.4 million SHA-512 hashes per second. RC4, a deprecated cipher that still shows up in legacy systems, allows 173,000 key recoveries per second. But Argon2id, an intentionally slow, memory-hard hash, drops that same GPU to roughly 2 attempts per second. That's not a typo — two. The difference between SHA-512 and Argon2id on the same hardware is over three million to one. This is why algorithm selection matters. A password database hashed with SHA-512 can be cracked in hours. The same database hashed with Argon2id would take centuries.


SLIDE 3 — Two Categories of Hash Functions
-------------------------------------------
All hash functions fall into one of two categories. Fast hashes — SHA-256, SHA-3, SHA-512, BLAKE3 — are engineered for maximum throughput. They produce billions of hashes per second on GPUs, generate deterministic fixed-length output, and have no tunable cost parameter. That's by design — they're meant for integrity verification, not secrecy. Slow hashes — Argon2id, bcrypt, scrypt, PBKDF2 — are the opposite. They're intentionally expensive to compute, with tunable work factors you can increase as hardware gets faster. The memory-hard variants like Argon2id and scrypt add another layer by requiring large amounts of RAM per hash, which starves GPUs of the parallelism they depend on. The key insight: these aren't better or worse than each other — they serve fundamentally different purposes.


SLIDE 4 — When to Use Fast Hashes
----------------------------------
Fast hashes belong in six common scenarios — all cases where speed is the priority and the input isn't a secret. File integrity: BLAKE3 hashes a 6.6 gigabyte file in under two seconds. Digital signatures: you hash the document before signing with RSA or ECDSA, and speed enables bulk verification. TLS and HTTPS: HMAC-SHA256 authenticates every packet in transit and must run at wire speed — you can't afford 250 milliseconds per packet. Data deduplication: identifying duplicate blocks in storage is a throughput problem, not a secrecy problem. API authentication: HMAC-based tokens like JWTs and webhook signatures need fast verification on every request. And checksums and caching: content-addressable storage, ETags, and cache keys. In none of these cases is the input a low-entropy secret, so there's no reason to pay the cost of a slow hash.


SLIDE 5 — When to Use Slow Hashes
----------------------------------
Now the other side. Any time a low-entropy secret — a password, PIN, or passphrase — must be stored or verified, you need a slow hash. The orange banner states the rule: if it came from a human and must be stored, use a slow, memory-hard hash. The three primary use cases are password storage for login credentials, key derivation from passphrases for disk encryption or wallet keys, and token or PIN storage for short codes, backup codes, and API secrets derived from human input. The bottom line shows why: the goal is to make each guess cost about 250 milliseconds. At that rate, an attacker trying 10 billion passwords — which covers every common password and many uncommon ones — would need approximately 79 years. That's the math that makes slow hashing work.


SLIDE 6 — Slow Hash Preference Order
-------------------------------------
Not all slow hashes are equal, and OWASP gives us a clear ranking. Number one is Argon2id — it's memory-hard, GPU-resistant, and protects against side-channel attacks. You can tune memory, iterations, and parallelism independently. Number two is scrypt — also memory-hard with strong GPU resistance, but it has three parameters that are harder to configure correctly. Set N too low and you silently lose your protection. It also lacks side-channel resistance. Number three is bcrypt — the workhorse since 1999, battle-tested with a simple single cost parameter. But it only uses 4 KB of RAM, which fits in GPU cache, making it increasingly vulnerable as GPUs improve. It also has a 72-byte input limit. Number four is PBKDF2 — use this only when FIPS-140 compliance is legally required. It's highly GPU-parallelizable, giving attackers a 10 to 30x speed advantage, and OWASP requires a minimum of 600,000 iterations.


SLIDE 7 — What Memory-Hard and CPU-Hard Mean
---------------------------------------------
This slide defines the terms from the previous one. Memory-hard means each hash requires a large block of RAM — 16 to 64 megabytes for Argon2id. A GPU like the RTX 4090 has 24 gigabytes of VRAM, but if each hash needs 64 megabytes, you can only run about 375 in parallel — not millions. Doubling the memory requirement quadruples the attacker's cost because it halves their thread count while doubling the work per thread. CPU-hard means many cycles but almost no RAM — bcrypt uses just 4 kilobytes. GPUs have thousands of cores, so they can parallelize CPU-hard work efficiently, giving attackers a 10 to 30x speed advantage. "Neither" means no deliberate cost at all — SHA-256, SHA-512, MD5. GPUs process billions per second. One GPU can exhaust common password lists in minutes. The bottom line: memory-hardness is the key differentiator. It makes the attacker's memory bandwidth the bottleneck, not raw compute power.


SLIDE 8 — Secure Memory Wiping by Language
-------------------------------------------
This is the part most presentations skip. Even with perfect Argon2id parameters, the plaintext password exists in memory while your application reads it from the user, passes it to the hashing library, and waits for garbage collection. If an attacker can dump process memory — through Heartbleed, a cold-boot attack, or local privilege escalation — they grab passwords that were never wiped. Rust and C/C++ can guarantee wiping. Rust's zeroize crate uses the Drop trait so secrets are zeroed when they leave scope, and the compiler cannot optimize it away. C/C++ needs memset_s — plain memset can be silently removed. Go, C#, Java, and Swift are partial — garbage collectors copy objects before you can wipe them. Python, JavaScript, and Ruby cannot do it at all — immutable strings and non-deterministic garbage collection. This is why every serious cryptographic library is written in Rust or C, even when the surrounding application is in Python or Node.


SLIDE 9 — Common Mistakes
--------------------------
Four mistakes we see repeatedly in production code. First, using a fast hash like SHA-256 or SHA-512 for password storage. GPUs crack hundreds of millions of these per second — it's essentially plaintext with extra steps. Second, using a slow hash like bcrypt to checksum files. It wastes CPU and degrades throughput for absolutely no security benefit — the input isn't secret. Third, storing hashes without unique salts. Even a slow hash without a per-user salt is vulnerable to precomputed rainbow tables. An attacker builds the table once and cracks every identical password instantly. Fourth, setting work factors once and never updating them. What was slow in 2015 is fast today. Hardware improves every year, and your cost parameters need to keep pace. The quote at the bottom captures it: speed is a feature for integrity; it is a vulnerability for passwords.


SLIDE 10 — Key Takeaways
-------------------------
Right hash, right job — that's the one sentence to remember. Fast hashes like SHA-256 and BLAKE3 for integrity, signatures, and authentication tokens. Slow hashes like Argon2id and bcrypt for passwords, PINs, and passphrase-derived keys. Never use a fast hash to store passwords — GPUs make it trivial. Never use a slow hash for bulk data — it wastes resources for no gain. And the often-forgotten fifth point: tune your work factors to target 250 milliseconds today, and plan to increase them over time. Hardware gets faster every year. Your hashing configuration should keep pace. If you take one action item from this talk, audit your current password hashing and check whether you're using Argon2id with OWASP-recommended parameters.
